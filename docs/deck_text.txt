1 · WHY THIS PROBLEM EXISTS
The KV-cache memory wall
During generation, an LLM caches a Key and Value vector for every token, in every layer and head — so it never recomputes the past.
The cache grows linearly with context length and batch size. It quickly outgrows the model weights themselves.
LLaMA-2 70B serving a 2048-token context at batch 32 needs ~640 GB of KV-cache. The GPU it runs on has 80 GB.
The papers' answers: evict unimportant tokens (H2O), compress the rest (CXL-SpecKV), or move the cache off-GPU (CXL).

3 · ARCHITECTURE
Store-and-forward: load → process → drain
YOUR PC — PYTHON HOST
distilgpt2 forward pass
attention → importance per token
FP16 → INT8 quantize (one scale)
frame + checksum over UART
UART
921,600 baud
⬄
BASYS 3 — YOUR VERILOG, 100 MHz
slice BRAM 32 KB   512 entries × 64 B + importance
evict filter   keep iff importance ≥ threshold
delta → zero-run RLE   the compression pipeline
bypass check   if bigger than raw → send raw
out BRAM 36 KB + stats   bitmap + stream + cycle counter
Why store-and-forward: the board's USB-UART chip has no flow-control wires, so streaming would drop bytes. Loading first also means the cycle counter brackets pure processing — engine speed and cable speed never blur.

10 · FINDINGS & COST
Honest findings, and the silicon receipt
Eviction does all the work
Delta+RLE bypassed on 100% of real INT8 vectors (paper reports ~30%). Without the bypass cap, “compression” would have grown the cache by 2%.
The fabric is fine — the data isn't compressible
The same pipeline compresses synthetic constants 21×. Quantized attention tensors simply have no zero-runs. A negative result, measured and explained.
Deep layers concentrate, shallow layers don't
Confirmed across 72 heads. This argues for per-layer eviction budgets — the Ada-KV / TailorKV research direction.
BRAM dominates because whole slices live on-chip — the memory-problem shape, visible in the utilization report.
